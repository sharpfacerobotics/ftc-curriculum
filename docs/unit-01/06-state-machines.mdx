# Lesson 6: State Machines

---

## Explanation

### The Problem with Simple If/Else

Imagine you want your robot's arm to cycle through three positions each time you press a button: down → up → score → down → up → score...

With simple if/else, you might try:

```java
if (gamepad1.a) {
    arm.setPosition(0.0); // down
} else if (gamepad1.a) {  // this never runs!
    arm.setPosition(0.5); // up
}
```

This doesn't work — both conditions check `gamepad1.a`, so the second one is unreachable. You need a way to **remember where you are** in a sequence. That's what a state machine does.

---

### What is a State Machine?

A **state machine** is a pattern where your code tracks a **current state** and transitions between states based on events (like button presses). At any moment, the robot is in exactly one state, and each state defines what the robot should be doing.

Think of a traffic light: it's always in one state (Red, Yellow, or Green), and it transitions to the next state on a timer. Your robot works the same way.

---

### Implementing with an `enum`

Java's `enum` (enumeration) is perfect for defining a fixed set of named states:

```java
enum ArmState {
    DOWN,
    UP,
    SCORE
}
```

Then declare a variable to track the current state:

```java
ArmState armState = ArmState.DOWN;
```

---

### A Full State Machine Example

```java
enum ArmState { DOWN, UP, SCORE }
ArmState armState = ArmState.DOWN;
boolean lastButtonA = false;  // tracks previous button state

@Override
public void runOpMode() {
    // ... hardware init ...
    waitForStart();

    while (opModeIsActive()) {
        boolean currentButtonA = gamepad1.a;

        // Only transition on a fresh button press (not while held)
        if (currentButtonA && !lastButtonA) {
            switch (armState) {
                case DOWN:
                    armState = ArmState.UP;
                    break;
                case UP:
                    armState = ArmState.SCORE;
                    break;
                case SCORE:
                    armState = ArmState.DOWN;
                    break;
            }
        }

        // Act based on current state
        switch (armState) {
            case DOWN:
                armServo.setPosition(0.0);
                break;
            case UP:
                armServo.setPosition(0.5);
                break;
            case SCORE:
                armServo.setPosition(1.0);
                break;
        }

        lastButtonA = currentButtonA;
        telemetry.addData("Arm State", armState);
        telemetry.update();
    }
}
```

---

### Edge Detection — The `lastButton` Pattern

Notice the `lastButtonA` variable above. This is called **edge detection** and it's essential for button-triggered state machines.

Without it, holding the button for even one extra loop cycle (which runs many times per second) would fire the state transition dozens of times instantly.

```java
// WRONG — fires continuously while held
if (gamepad1.a) { transition(); }

// RIGHT — fires only on the frame the button is first pressed
if (gamepad1.a && !lastButtonA) { transition(); }
lastButtonA = gamepad1.a;
```

The logic: `gamepad1.a` is true AND `lastButtonA` was false — meaning the button just became pressed this frame. Next frame, `lastButtonA` will also be true, so the condition fails until the button is released and pressed again.

---

### The `switch` Statement

A `switch` statement is a cleaner alternative to long `if / else if` chains when checking a single variable against multiple values:

```java
switch (armState) {
    case DOWN:
        // do something
        break;
    case UP:
        // do something else
        break;
    default:
        // runs if no case matched
        break;
}
```

:::warning Don't forget `break`
Without `break` at the end of each case, Java **falls through** to the next case automatically. This is almost never what you want and causes very hard-to-find bugs.
:::

## Quiz

**Q1. What problem does a state machine solve that simple if/else cannot?**

<details>
<summary>Show answer</summary>

A state machine remembers where the robot is in a sequence. Simple if/else only reacts to what's happening right now — it has no memory of past events.

</details>

---

**Q2. What is edge detection and why is it needed?**

<details>
<summary>Show answer</summary>

Edge detection tracks the **transition** of a button from not-pressed to pressed (a "rising edge"). It's needed because the OpMode loop runs hundreds of times per second — without edge detection, holding a button for even a fraction of a second triggers the action dozens of times.

</details>

---

**Q3. What happens if you forget `break` in a `switch` statement?**

<details>
<summary>Show answer</summary>

Java "falls through" to the next case and executes it too, continuing until it hits a `break` or the end of the switch. This means multiple cases run when you only intended one.

</details>

---

**Q4. Why is `enum` a good choice for defining states?**

<details>
<summary>Show answer</summary>

An `enum` creates a fixed, named set of values. This prevents typos (you can't accidentally assign an invalid state), makes the code self-documenting, and allows the compiler to warn you if you miss a case in a `switch` statement.

</details>
